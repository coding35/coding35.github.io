Based on the Sidecar pattern, the Ambassador is an application that runs alongside client applications to provide additional connectivity functionality such as routing, logging, monitoring, authentication, resilience and more.

In short, it is a proxy that sits between the client application and the back end services. It takes requests from a client, performs some action based on the request, and then forwards the request to the appropriate backend service. This process is known as request routing and while that is beneficial on its own, another benefit is it can also be used to perform the additional functionality as mentioned previously.

Microsoft documentation states

"An ambassador service can be thought of as an out-of-process proxy which is co-located with the client. This pattern can be useful for offloading common client connectivity tasks such as monitoring, logging, routing, security (such as TLS), and resiliency patterns in a language agnostic way. It is often used with legacy applications, or other applications that are difficult to modify, to extend their networking capabilities. It can also enable a specialized team to implement those features." Ambassador Pattern Microsoft

A common use case is to use the Ambassador to extend the lifetime of legacy applications where it may not be cost effective or feasible.

For Example

During the Y2K scare, many companies were faced with the problem of migrating or modernizing their legacy applications in order to ensure that they would continue to function after the year 2000. The problem was that many of these legacy applications were written in COBOL running on mainframes that were no longer supported by the vendor. The cost of migrating these applications was a major concern. One solution was to use the Ambassador to extend the lifetime of the legacy applications by co-locating the legacy application with the Ambassador whereby the Ambassador would manage date conversions and thereby allowing the legacy application to continue operations until they could be patched or migrated to a new platform.

Considering the previous example we can break down the use case:

One or more legacy applications need additional connectivity functionality and are longer supported by the vendor or would require significant effort to upgrade. These applications are still in use and are critical to the business. One option is to migrate the application to a new server or cloud platform but that may be costly and time consuming. The other option is to use the Ambassador to extend the lifetime of the legacy application by co-locating the legacy application with the Ambassador and thereby allowing the legacy application to continue to run on the old server with the additional connectivity functionality provided by the Ambassador.

When to use the Ambassador pattern?

  • One or more legacy applications needing common or cross-cutting connectivity features.
  • There are shared connectivity concerns that can be offloaded to a separate pipeline.
  • Cannot easily modify or migrate existing applications.
  • A low cost alternative is needed "wrap" existing applications with modern features.

When not to use the Ambassador pattern?

  • Latency concerns. The Ambassador providing additional connectivity concerns will introduce latency in the request.
  • All legacy applications share a common language and framework. In this scenario a better approach would be to develop a package or module that could be added to each application
  • There are no cross-cutting network connectivity concerns.
